## Why

CSV reports generated by the analysis engine use static filenames (e.g., `NestingDepth.csv`, `CyclomaticComplexity.csv`). Because reports are generated per-module inside loop iterations, each module overwrites the report file from the previous module. Only the last module's data survives, making per-module analysis impossible.

## What Changes

- Remove static filename writes from `CsvReportWriter` and all callers
- For each report type, create a subdirectory named after the report (e.g., `NestingDepth/`)
- Write each per-module report into that directory with filename = report name + file extension
- Ensure report type directory is created automatically if missing
- Apply the same pattern to SBOM report files in `SBOMTool`

## Capabilities

### New Capabilities
- `report-directory-structure`: Centralized CSV report output strategy that organizes per-module reports into report-type-named directories

### Modified Capabilities
None — this is a structural change to output layout, not a change to analysis requirements.

## Impact

- **`CsvReportWriter.java`**: Change signature to accept a path component for the subdirectory; add `createDirectories` logic
- **`JavaTool.java`**: Update all 22 `CsvReportWriter.*` calls to pass the report-name-based path instead of raw `workingDirectory`
- **`SBOMTool.java`**: Update 2 CSV report calls to use new path structure
- **No impact** on schemas, prompts, task YAML, or LLM interaction — purely a Java-side output reorganization
